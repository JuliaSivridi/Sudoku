1. Overview

Stler Sudoku is a personal Android application built entirely with Kotlin and Jetpack Compose. It runs fully offline — no network calls, no accounts, no analytics. All state is persisted locally via SharedPreferences.

Key Design Goals

  • Relaxing UX — no timers, no penalties, no pressure
  • Two input modes — Number first (digit → cell) and Cell first (cell → digit), switchable at any time
  • Full undo history — every move is reversible back to the start of the game
  • Adaptive theming — four accent palettes, dark/light mode follows system setting
  • Unique puzzles — generator guarantees exactly one solution per puzzle

Data Flow

  1. App start → MainActivity creates ThemeViewModel and InputPreferenceViewModel, reads persisted values, wraps content in SudokuTheme
  2. Navigation → AppNavigation owns GameViewModel and SolverViewModel; passes inputPreference to GameScreen
  3. Game mutation → composable calls ViewModel method → ViewModel updates _state: MutableStateFlow<GameState>
  4. Recomposition → collectAsState() in screen triggers UI update
  5. Auto-save → init block launches state.collect { … }; saves to SharedPreferences after every state change

5. State Management

All mutable state lives in AndroidViewModel subclasses exposed as StateFlow. Composables collect state with collectAsState().

Auto-save Pattern

The save happens inside a StateFlow collector launched in GameViewModel.init, ensuring it fires synchronously with every state emission — even if Android kills the process shortly after:

init {
    viewModelScope.launch {
        state.collect { gameState ->
            when {
                gameState.isComplete -> {
                    saveManager.clearSavedGame()
                    if (!statsRecorded) {
                        statsRecorded = true
                        statsManager.increment(gameState.difficulty)
                    }
                }
                gameState.board.any { row -> row.any { !it.isGiven && it.value != 0 } } ->
                    saveManager.saveGame(gameState)
            }
        }
    }
}

6. Game Logic — Generator & Solver

SudokuSolver

A pure Kotlin object with three functions used by both generation and the Solver screen:

FunctionDescription
isValid(board, row, col, num)Checks row, column and 3×3 box for conflicts
solve(board)In-place backtracking solver; returns true if solved
countSolutions(board, limit=2)Counts solutions up to limit; stops early — used to verify uniqueness

SudokuGenerator — Puzzle Generation

Two-phase algorithm:

Phase 1 — Full board generation

Recursive backtracking with randomised digit order at each cell. For each empty cell, the digits 1–9 are shuffled before trying, producing a different valid complete board on every call.

Phase 2 — Cell removal

Cells are removed in random order. After each removal, SudokuSolver.countSolutions() is called with limit = 2. If the count is still 1 (unique solution), the cell stays empty. Otherwise it is restored. Removal continues until the target number of empty cells is reached:

val cellsToRemove = 81 - difficulty.givens
// EASY:   81 - 46 = 35 cells removed
// MEDIUM: 81 - 36 = 45 cells removed
// HARD:   81 - 29 = 52 cells removed
Uniqueness guarantee: Every generated puzzle has exactly one solution. The countSolutions early-exit at 2 keeps the check fast — worst case is still exponential but terminates as soon as a second solution branch is found.

Conflict Detection (hasConflict)

Implemented in SudokuGrid.kt as a private function. A cell is highlighted red when:

  • It is not a given cell (isGiven == false)
  • Its value matches any other cell in the same row, column, or 3×3 box (given or user-placed)

Given cells are never highlighted — the early if (cell.isGiven) return false prevents it.

Auto-notes (Clues)

When Clues is toggled on, every empty cell's valid candidates are computed on the fly:

val possible = (1..9).filter { digit ->
    canPlace(current.board, row, col, digit)
}.toSet()
cell.copy(notes = possible)

canPlace checks row, column and box — identical logic to SudokuSolver.isValid. Toggling off maps all cells to cell.copy(notes = emptySet()). Both operations are pushed to the undo stack. The autoNotesActive flag is intentionally not reset by undo.

8. UI Components

SudokuGrid

A Box with aspectRatio(1f). Cell backgrounds are composable layers; grid lines are drawn on top via a drawWithContent Canvas overlay, ensuring borders are never obscured by cell fills.

Cell background priority (layered)

  1. Row/column highlight (CellHighlight*) if cell is in the same row or column as selectedCell
  2. Digit match highlight (cellDigitHighlight*, ~20–50% alpha accent) if cell value equals selectedDigit
  3. Selected cell (CellSelected*) if this is selectedCell
  4. Conflict background (ConflictBg*, ~20–50% alpha red) layered on top of any of the above

Grid border rendering

Border typeStrokeColor
Outer frame2 dpGridOuterBorder*
3×3 box boundaries (i % 3 == 0)2 dpGridOuterBorder*
Inner cell lines1 dpGridInnerBorder*

Notes mini-grid

When a cell has notes and no value, a fixed 3×3 Column/Row is rendered. Digit N occupies position row = (N-1)/3, col = (N-1)%3 — so 1 is always top-left and 9 always bottom-right, regardless of which notes are present.

NumberRow

A 3×3 grid of digit cells. Each cell:

  • Width = 1/3 of total width (matching a 3×3 box in the grid above)
  • aspectRatio(2f) — height is half the width (~1.5× the old single-row height)
  • Font size: 32 sp
  • Disappears when all 9 of that digit are placed (digitCounts[digit] >= 9)
  • Selection highlight suppressed in Cell first mode (showSelection parameter)

ControlButtons

Five equal-weight columns in a Row. Each button is a full-width Column (icon + label) — the clickable area covers the entire column width, avoiding missed taps.

10. Persistence

All persistence uses a single SharedPreferences file named "sudoku_prefs".

ManagerKeyFormat
GameSaveManager"saved_game"JSON string (see below)
StatsManager"stats_{DIFFICULTY}"Integer count per difficulty name
ThemeManager"color_theme"AppColorTheme.name string
InputPreferenceManager"input_preference"InputPreference.name string

GameSaveManager — JSON Schema

{
  "difficulty": "MEDIUM",
  "board": [                        // 9×9 array
    [ {"v": 5, "g": true,  "n": []},
      {"v": 0, "g": false, "n": [1, 3, 7]},
      ... ],
    ...
  ],
  "solution": [                     // 9×9 array of ints
    [5, 3, 4, 6, 7, 8, 9, 1, 2],
    ...
  ]
}

Cell fields: v = value (0 = empty), g = isGiven, n = notes array. The undo stack and UI state (selectedCell, selectedDigit, inputMode) are not saved — the game resumes in a clean NORMAL state.

Auto-save trigger

Saving happens inside a StateFlow collector in GameViewModel.init (not in onCleared(), which is unreliable when Android kills the process from the recents screen). The game is saved whenever at least one non-given cell has a value. On completion, the save is cleared and stats are incremented exactly once per game (guarded by the statsRecorded flag).

11. Settings & Preferences

Theme

ThemeViewModel holds a MutableStateFlow<AppColorTheme> initialised from ThemeManager. MainActivity collects it and passes the value to SudokuTheme, triggering a full recomposition — the accent color changes immediately without restarting the app.

Input Preference

InputPreferenceViewModel follows the same pattern. The value is collected in AppNavigation and passed as a parameter to GameScreen, where it routes cell-tap and digit-tap callbacks to different ViewModel methods:

PreferenceDigit pad highlightNotes multi-tapClear button
Number firstShows selected digitMust re-select cell each timeToggles ERASE mode
Cell firstHiddenSelect cell once, tap digits freelyDirect erase action
Default: Number first. New users familiar with classic Sudoku apps will find the default behaviour matches their expectation. Cell first is recommended for players who prefer filling multiple notes in one cell without switching focus.
